How Claude Code Works

Claude Code is Anthropic's agentic coding tool that lives in your terminal. It understands your codebase, executes tasks, and helps you code faster through natural language. This page documents the architecture of the official plugins repository — the extensibility ecosystem that powers commands, agents, skills, hooks, and MCP integrations.

This repository contains 12 official plugins, a plugin marketplace registry, and extensive examples. The core CLI binary is separate — plugins extend it through a well-defined contract.

System Architecture Layers

Claude Code is organized into stacked layers, from user input at the top to tool execution at the bottom.

End-to-End Architecture
User Input Layer /commit, /feature-dev, plain prompts, CLI flags
Command Resolution Layer Plugin discovery, command loading, context injection
Hook Interception Layer SessionStart, PreToolUse, UserPromptSubmit, Stop
Claude Model & Reasoning Layer Multi-model (Opus, Sonnet, Haiku) — agents, skills, tool calls
Tool Execution Layer Bash, Read, Write, Edit, Glob, Grep, WebFetch, Skill…
Post-Tool & State Management PostToolUse hooks, rule engine, session state, output

Commands

Commands are markdown files with YAML frontmatter that define slash commands (/commit, /feature-dev, etc.). They are the primary user-facing extension point.

Command File Format

---
description: Create a git commit
argument-hint: Optional commit message
allowed-tools: Bash(git add:*), Bash(git status:*), Bash(git commit:*)
---

# Command body in Markdown

## Context
- Current git status: !`git status`
- Current diff: !`git diff HEAD`
- Recent commits: !`git log --oneline -10`

## Your task
Based on the above changes, create a single git commit.

Key Features

  • allowed-tools — Restricts which tools the command can use (security boundary)
  • Context injection — !`command` syntax dynamically runs shell commands and injects output
  • $ARGUMENTS — Passes user arguments into the command body
  • argument-hint — Tab-completion hint shown in CLI
Example: /commit-push-pr Command

The /commit-push-pr command chains three git operations. It:

  1. Reads staged/unstaged changes via context injection
  2. Creates a commit with a generated message
  3. Pushes to the remote branch
  4. Creates a pull request using gh pr create

Allowed tools: Bash(git:*), Bash(gh:*)

Hook System

Hooks are event-driven interceptors that run at specific points in the CLI lifecycle. They can validate, warn, block, or inject context.

Hook Events

EventWhenCan Block?
SessionStartSession beginsNo — injects context
PreToolUseBefore any tool executesYes (exit code 2)
PostToolUseAfter tool completesNo
UserPromptSubmitUser submits a promptNo
StopClaude wants to exitYes (can force continue)
SubagentStopSubagent stopsYes
SessionEndSession endsNo
PreCompactBefore context compressionNo
StopFailureTurn ends due to API errorNo
NotificationNotification eventNo

Hook Configuration (hooks.json)

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash|Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "python3 ${CLAUDE_PLUGIN_ROOT}/hooks/validate.py",
            "timeout": 10
          }
        ]
      }
    ]
  }
}

Hook I/O Protocol

Input (stdin JSON)

{
  "session_id": "abc123",
  "hook_event_name": "PreToolUse",
  "tool_name": "Write",
  "tool_input": {
    "file_path": "src/app.ts",
    "content": "..."
  }
}

Output (stdout JSON)

{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny"
  },
  "systemMessage": "Warning!"
}
// Exit code: 0 = allow, 2 = block
Hookify Rule Engine

The hookify plugin provides a sophisticated rule engine that evaluates user-defined rules from markdown files.

Simple Rule Format

---
name: block-dangerous-rm
enabled: true
event: bash
pattern: rm\s+-rf
action: block
---

Warning! Dangerous rm command detected.

Advanced Rule Format (Multiple Conditions)

---
name: protect-env-files
conditions:
  - field: file_path
    operator: regex_match
    pattern: \.env$
  - field: new_text
    operator: contains
    pattern: API_KEY
action: block
---

Operators: regex_match, contains, equals, not_contains, not_equals

Fields: command (Bash), file_path, new_text, old_text, content, reason, transcript

Tool System

Tools are the actions Claude can take. Each tool receives structured JSON input and returns output. Commands can restrict which tools are available via allowed-tools.

Core Tools

Bash
Execute shell commands. Input: {"command": "..."}. Supports timeouts, background execution.
Read
Read file contents. Supports line ranges, images, PDFs, Jupyter notebooks.
Write
Create or overwrite files. Input: {"file_path": "...", "content": "..."}
Edit
Exact string replacements. Input: {"file_path", "old_string", "new_string"}
Glob
File pattern matching. Find files by name patterns like **/*.ts.
Grep
Content search with regex. Built on ripgrep. Supports context lines, file type filters.
WebFetch
Fetch and analyze web content. Converts HTML to markdown, processes with AI.
WebSearch
Search the web for up-to-date information. Returns formatted results with links.
TodoWrite
Structured task list management. Track progress with pending/in_progress/completed states.
Agent
Launch specialized subagents. Supports parallel execution, background running, worktree isolation.
Skill
Invoke plugin skills. Loads knowledge units into the current context.
NotebookEdit
Edit Jupyter notebook cells. Supports replace, insert, and delete operations.

Permission Control

The allowed-tools field in command frontmatter restricts tool access. Pattern-based matching:

# Only allow specific git and GitHub CLI commands
allowed-tools: Bash(git add:*), Bash(git commit:*), Bash(gh pr create:*)

# Allow read-only tools plus specific MCP tool
allowed-tools: Glob, Grep, Read, mcp__github__create_inline_comment

Command Execution Flow

What happens when a user types /commit in the terminal.

End-to-End: /commit Execution
1. User types /commit
2. CLI loads plugins/commit-commands/commands/commit.md
3. Context injection: !`git status`, !`git diff HEAD`, !`git log`
4. Claude receives context + instructions, generates tool calls
5. PreToolUse hook validates: allowed-tools check + security patterns
6. Tools execute: git add, git commit -m "..."
7. PostToolUse hook processes output → result shown to user

Feature Development Workflow

The /feature-dev command implements a structured 7-phase development process.

7-Phase Feature Development
Phase 1: Discovery Understand what needs to be built. Clarify requirements.
Phase 2: Codebase Exploration Launch 2-3 code-explorer agents in parallel
Phase 3: Clarifying Questions Identify all ambiguities, edge cases. CRITICAL — do not skip.
Phase 4: Architecture Design Launch 2-3 code-architect agents with different approaches
Phase 5: User Choice Present approaches with trade-offs. User picks direction.
Phase 6: Implementation Code the feature using chosen architecture
Phase 7: Review & Completion Launch code-reviewer agents. Final quality check.

Agent Roles

code-explorer
Sonnet model. Traces architecture, finds similar features, identifies patterns. Returns key file lists.
code-architect
Sonnet model. Designs implementation approaches: minimal, clean, or pragmatic balance.
code-reviewer
Reviews implementation quality, correctness, and adherence to patterns.

Code Review Pipeline

The /code-review command orchestrates multiple agents at different model tiers for PR review.

Multi-Agent Code Review Pipeline
Step 1: Haiku agent checks PR status (closed? draft? already reviewed?)
Step 2: Haiku agent finds relevant CLAUDE.md files
Step 3: Sonnet agent summarizes PR changes
Step 4: Launch 4 review agents in parallel
Sonnet Agent 1
CLAUDE.md compliance audit (part 1)
Sonnet Agent 2
CLAUDE.md compliance audit (part 2)
Opus Agent 3
Bug scanning (diff-only, no context)
Opus Agent 4
Security + logic issues (changed code only)
Step 5: Validation subagents verify each issue (Opus for bugs, Sonnet for CLAUDE.md)
Step 6: Filter unvalidated issues → HIGH SIGNAL only
Step 7-9: Output summary → Post inline comments with gh pr comment
Only HIGH SIGNAL issues are reported: compilation failures, clear logic errors, and unambiguous CLAUDE.md violations. False positives erode trust — when in doubt, don't flag it.

Settings & Configuration

Permissive settings. Fewer prompts. For experienced users who trust their workflow.
