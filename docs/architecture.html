How Claude Code Works

Claude Code is Anthropic's agentic coding tool that lives in your terminal. This page documents the architecture of the official plugins repository — 13 production plugins, 15 commands, 15 agents, 11 skills, and a sophisticated hook system.

This repository is the plugin ecosystem, not the core CLI binary. Plugins extend Claude Code through a well-defined contract of commands, agents, skills, hooks, and MCP servers.

System Architecture Layers

Every request flows through these layers, from user input to tool execution and back.

End-to-End Architecture — Top to Bottom
1. User Input/commit, /feature-dev, plain prompts, CLI flags
2. Plugin & Command Resolutionmarketplace.json → plugin.json → command .md loaded
3. Context Injection!`git status`, !`git diff`, $ARGUMENTS substituted
4. Hook Interception (Pre)SessionStart, UserPromptSubmit hooks fire
5. Claude Model & ReasoningMulti-model (Opus/Sonnet/Haiku), agent spawning, tool calls
6. PreToolUse HooksSecurity checks, hookify rules, permission validation
7. Tool ExecutionBash, Read, Write, Edit, Glob, Grep, WebFetch, Agent, Skill
8. PostToolUse HooksResult processing, rule evaluation, session state updates
9. Stop / Continue DecisionStop hooks evaluate — may force continuation

Data Flow: Hook Pipeline (Detailed)

Every hook follows the same I/O protocol: JSON on stdin, JSON+exit code on stdout.

Hook Execution Protocol

Input (JSON via stdin)

{
  "session_id": "abc123",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": {
    "command": "rm -rf /tmp/test"
  }
}

Output (JSON via stdout)

{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny"
  },
  "systemMessage": "Blocked!"
}
// Exit code: 0=allow, 2=block
Hookify Rule Engine — Internal Call Chain
hooks.json registers: python3 pretooluse.py
↓ CLI spawns subprocess, pipes JSON to stdin
pretooluse.py (main)
1. Reads JSON from stdin
2. Maps tool_name → event type (Bash→"bash", Edit/Write→"file")
↓ calls config_loader.load_rules(event="bash")
config_loader.py → load_rules()
1. Globs .claude/hookify.*.local.md files
2. For each: extract_frontmatter() → parse YAML
3. Rule.from_dict() → builds Rule with Conditions
4. Filters by event + enabled flag
↓ returns List[Rule]
rule_engine.py → RuleEngine.evaluate_rules()
1. For each rule: _rule_matches() checks all conditions
2. _extract_field() gets value from tool_input
3. _check_condition() applies operator (regex_match, contains, equals…)
4. Separates blocking_rules vs warning_rules
↓ returns {systemMessage, hookSpecificOutput}
pretooluse.py outputs JSON to stdout + exit(0)

Ralph Wiggum: Iteration Loop State Machine

The ralph-wiggum plugin implements a self-referential iteration loop using a Stop hook and a state file. This lets Claude re-attempt a task multiple times before stopping.

State Machine — Ralph Loop Lifecycle
User: /ralph-loop "Implement feature X" --max-iterations 5
↓ scripts/setup-ralph-loop.sh creates state file
State File Created: .claude/ralph-loop.local.md
YAML frontmatter: iteration: 1, max_iterations: 5, completion_promise: null
Body: Original prompt text
↓ Claude works on task, calls tools, makes progress
Claude decides to stop (end_turn)
↓ Stop hook fires: stop-hook.sh
Stop Hook Evaluates:
1. Reads .claude/ralph-loop.local.md
2. Parses iteration + max_iterations
3. If completion_promise set → checks transcript for <promise> tag
4. If iteration < max_iterations → BLOCK STOP
↓ decision: "block", reason: ORIGINAL_PROMPT re-injected
State File Updated: iteration: 2
Claude receives original prompt again, sees previous work in files
↓ loop repeats until max_iterations or promise fulfilled
Iteration ≥ max_iterations
State file deleted
exit(0) — stop allowed
Promise tag found in transcript
State file deleted
exit(0) — stop allowed

State File Format

---
iteration: 1
max_iterations: 5
completion_promise: "All tests pass"  # optional
---

Implement feature X with the following requirements...
# ↑ This body is re-injected as the "reason" when stop is blocked

Cancellation

/cancel-ralph deletes the state file, which causes the stop hook to find no state and allow exit normally.

Evaluation Algorithm

def evaluate_rules(rules, input_data):
    blocking_rules = []
    warning_rules = []

    for rule in rules:
        if _rule_matches(rule, input_data):   # ALL conditions must match (AND)
            if rule.action == "block":
                blocking_rules.append(rule)
            else:
                warning_rules.append(rule)

    # Blocking rules take priority
    if blocking_rules:
        return {"hookSpecificOutput": {"permissionDecision": "deny"},
                "systemMessage": combined_messages}
    elif warning_rules:
        return {"systemMessage": combined_messages}
    return {}  # no match — allow

Configuration & Settings

Settings Presets (from examples/settings/)

Strict Mode
Denies WebSearch+WebFetch. Requires approval for Bash. Disables bypass permissions. allowManagedHooksOnly + allowManagedPermissionRulesOnly.
Lax Mode
Minimal restrictions. Only disableBypassPermissionsMode: "disable" and empty strictKnownMarketplaces.
Bash Sandbox
sandbox.enabled: true. No unsandboxed commands. No local binding. No unix sockets. No allowed domains.
